# 术语表（Glossary）模板

翻译前**必读**。本文件帮助你建立**术语一致性**，避免同词异译造成混乱。

> ⚠️ 这是一个**模板**。下表的示例仅供格式参考，**翻译具体项目时请按本项目实际术语改写**：
> 把项目里的产品名、专有名词、易混淆的业务概念填进去，锁定译法。

分两类：
- **A 类 = 不翻译**（保持原文）：产品名、缩写、代码标识符、章节锚点
- **B 类 = 锁定译法**（按表统一）：有多种可能译法的业务概念词

---

## A 类：保持原文，绝不翻译

通用规则（任何项目都适用）：

| 类别 | 说明 | 示例（替换为你的项目实际术语）|
|------|------|------|
| 产品名 / 品牌名 | 大小写严格按官方写法 | `PostHog`（不写 Posthog）、`GitHub`（不写 Github） |
| 技术缩写 | 行业通用，不展开 | `API`、`SDK`、`HTTP`、`URL`、`JSON`、`CSS` |
| 模型/算法缩写 | 专业术语 | `RFM`、`LTV`、`A/B test`、`SEO` |
| 代码标识符 | 变量名、函数名、类名 | `user_id`、`getUserProfile()` |
| 文件路径 / 命令 | 路径与 CLI 命令 | `npm install`、`./config.yml` |
| 章节锚点 | 跨页面导航的编号标识 | `§01`、`Part 1`、`Chapter 2` |
| HTML/CSS 语法 | 标签名、属性、选择器 | `<div>`、`class="btn"`、`href` |

> 章节锚点（如 `§编号`、`Part N`、`Chapter N`）必须保留：它们是目录与跨页导航的标识符，翻译会破坏链接。

## 使用方法

1. **翻译前先填表**：扫描项目原文，把高频产品名（填 A 类）、业务概念词（填 B 类）录入
2. 翻译每批前，对照本表把涉及的术语圈出
3. 填译文时逐个套用对应语言的锁定译法
4. 同一批内若出现同词多次，**必须**用同一个译法
5. 若翻译过程中发现新的高频术语未列入本表，**先补表再继续**（避免风格漂移）

## 容易翻错的点（通用）

- 产品名大小写严格按官方：`GitHub` 不写 `Github`，`iPhone` 不写 `Iphone`
- 缩写词保持原样：`API` 不展开成 `Application Programming Interface`（除非上下文需要）
- 中英混排时，英文词前后留半角空格（排版规范）
- 同一术语在不同语言里译法一旦确定，全项目不可更改
