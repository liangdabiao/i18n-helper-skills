# 提取与跳过规则详解

`extract.py` 通过共享扫描器 `htmlscanner.py` 决定**哪些文本进翻译源、哪些跳过**。
翻译时同样要理解这些规则，避免对「应跳过的内容」强行翻译。
以下规则均在真实静态书站（PostHog 手册，36 页 HTML）上验证通过。

## ✅ 会提取的（需翻译）

扫描器对这些标签下的**可见文本节点**提取翻译条目：

| 标签 | 说明 |
|------|------|
| `title` | 浏览器标签页标题 |
| `h1`–`h6` | 各级标题 |
| `p` | 段落正文 |
| `li` | 列表项 |
| `td` `th` `caption` | 表格单元格、表头、表标题 |
| `a` | 链接文本（仅文本，不含 href） |
| `blockquote` | 引用块 |
| `strong` `em` `b` `i` | 强调文本（内联混排时单独成条） |
| `span` | span 内文本 |
| `figcaption` `label` `option` `button` | 图注、表单标签、选项、按钮 |

**属性值**（同样提取）：
- `img` / `area` 的 `alt`
- `input` 的 `placeholder`

## ❌ 绝不提取 / 绝不翻译的

扫描器**整段跳过**这些，翻译时也绝不能碰：

| 内容 | 原因 |
|------|------|
| `<pre> <code> <script> <style> <kbd> <samp>` 整块 | 代码/脚本/样式，改了会破坏功能 |
| URL（`https?://` 开头） | 链接地址，翻译会失效 |
| 百分号编码串（`%E7%AC%AC3%E8%AF%BE...`） | URL 内编码中文，是地址的一部分 |
| `href` / `src` 的路径值 | 资源引用 |
| CSS `class` / `id` | 样式选择器 |
| `lang` 属性值 | 由 apply.py 按目标语言统一改写 |
| 纯符号/数字串（`§01`、`※`、`←`、`→`、`·`） | 无语义，无需翻译 |
| 图片文件名（如 `image.png`） | 无语义 alt |

## 边界情况处理（重要）

以下情况在真实站点中常见，扫描器的处理方式如下：

### 1. 内联标签混排
```html
<p>3，<strong>原来的数据：</strong></p>
```
- `<strong>` 内文本单独成条（key: `...:strong:N`）
- `<p>` 的直接文本「3，」若全是符号/数字则跳过；否则单独成条
- **翻译时**：分别翻译每条，标签嵌套结构 apply 时自动保持

### 2. `<a><span>§01</span>标题文字</a>`（目录条目）
```html
<a href="01.html"><span class="toc-num">§01</span>PostHog 是什么</a>
```
- `§01`（纯符号）跳过
- `<a>` 的直接文本「PostHog 是什么」归属 `a` 标签，**会被提取**
- 这是「内联混合」最常见形态，扫描器通过「最近可翻译祖先」正确归属

### 3. `<title>` 双段式标题
```html
<title>§20 RFM 模型：给每个用户的价值打分 · PostHog 用户行为分析手册</title>
```
- 整体作为一条提取（含 `·` 分隔符）
- **翻译时**：整体翻译，但 `§20`、`PostHog` 等按 glossary 保留不译

### 4. 代码块内中文注释（默认跳过）
```html
<pre><code>user_id：用户身份
behavior_type：用户行为类型</code></pre>
```
- **整块跳过**，中文注释不提取（安全优先）
- 若用户**显式要求**翻译代码注释：需人工逐行确认，apply 不支持自动替换代码块内容

### 5. HTML 实体与转义
```html
<p>time_on_page &lt;= 180</p>
```
- `&lt;` `&gt;` `&amp;` 等实体在提取时 `convert_charrefs=False`，保持原样
- apply 替换时只动文本，实体逐字保留，**不会被破坏**

### 6. 跨文件重复（导航词）
```
「下一章 →」「← 上一章」「← 返回目录」「目录」
```
- 这些词在多个页面出现，扫描器通过后处理归并为 `common.*` key
- **翻译时**：每个 common key 只翻一次，全站所有页面自动统一
- **切勿**把不同页面的「下一章」翻成不同说法

### 7. 属性值 vs 文本节点（同名陷阱）
```html
<nav class="toc" aria-label="目录"><h2>目录</h2></nav>
```
- `aria-label="目录"`（属性值）和 `<h2>目录</h2>`（文本节点）字面相同，但是**两处**。
- 扫描器只提取**文本节点**（`h2` 的「目录」），**不提取** `aria-label` 这类通用属性值。
- apply.py 内置 `in_attr()` 检测：定位文本节点时，会**跳过**落在属性值 `"..."` 内的位置，
  避免把 `aria-label="目录"` 错当成 `h2` 的文本来替换。
- **结果**：`<h2>` 会正确翻译，`aria-label` 保持原文（无障碍属性，翻译与否不影响显示）。
  若需同步翻译 `aria-label`，目前需手动改（apply 不自动处理通用属性）。

### 8. `<div>` 直接含文本（横幅、标签等）
```html
<div class="toc-part">Part 1 · 认知——用户行为数据是什么</div>
```
- 扫描器**已把 `div` 列入可翻译标签**，这类直接含文本的 `div` 会被提取。
- 注意：很多 `div` 只是容器（文本在子标签里），扫描器只取 `div` 的**直接文本节点**，
  不会把子标签的文本重复归属。
- **翻译时**：与普通段落一样处理即可。

### 9. 弯引号 vs 直引号（翻译者易踩的坑）
原文常用**弯引号** `“ ”`（U+201C/201D），翻译者手敲 JSON 时可能误用**直引号** `"`（U+0022）：
```json
// 原文是 “维度灾难”，翻译者写成
"维度灾难": "the \"curse of dimensionality\""   // ← key 的引号变了，会漏匹配
```
- apply.py 内置**引号归一化兜底**：匹配前把 `“ ”` `‘ ’` 都映射为直引号，
  所以无论你用哪种引号，都能正确匹配原文。
- **建议**：翻译 JSON 时尽量**照抄原文的引号形式**，但即使敲错了也不会漏译（有兜底）。

### 10. 短文本嵌在长文本内（标题两形态）
```html
<!-- 完整 title（含手册后缀）-->
<title>§01 PostHog 是什么 · PostHog 用户行为分析手册</title>
<!-- 目录里的短标题（无后缀）-->
<a>§01 PostHog 是什么</a>
```
- 「短标题」是「完整标题」的子串，两者**都**会被提取为独立 key。
- apply.py 用「已占用区间」追踪：替换完整标题时，短标题若落在同一区间会被自动跳过，
  **不会产生「译文碎片重复」的乱码**（如 `...than GAined individual...`）。
- **翻译时**：两种形态都要翻，且译文要一致（短标题 = 完整标题去掉后缀部分）。

## 翻译时的自检清单

每批翻译完，逐条核对：
- [ ] 没有翻译任何 URL（`https://...`）
- [ ] 没有翻译代码块内容
- [ ] 技术术语与 glossary 一致（PostHog 不写成 Posthog，埋点统一译法）
- [ ] 保留了 `§编号`、`Part N`、HTML 实体
- [ ] common.* key 的译文在全站通用合理
- [ ] 短标题与完整标题的译文一致（见第 10 条）
- [ ] 引号尽量照抄原文（弯/直都行，有兜底但建议一致）
